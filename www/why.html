What Is Intercooler?

Intercooler is a Partial View Controller client-side framework: it communicates with the server via AJAX, but rather than expecting JSON responses it expects fragments of HTML (i.e. Partial Views), which are then loaded into the DOM directly.

This is in contrast with more traditional client side MVC frameworks such as Ember.js or Angular.js which communicate with the server using JSON requests and JSON responses.

Here is a diagram how a PVC framework works:

Note that the response content is a partial bit of HTML, and it is swapped in place of the target div.

Intercooler formalizes this approach, using HTML5-style attributes to drive the AJAX requests.

Other examples of PVC-like frameworks are Turbolinks and pjax.

Why would I use a PVC framework?

The primary advantages of PVC frameworks is their simplicity: they preserve many of the advantages, advances, tools and concepts of traditional server-side web applications and can often involve only slight modifications to existing applications to move over to.

This is in contrast to most Javascript MVC frameworks, which are very heavyweight. They give you a lot, but they require a lot as well: you must set up a client side model, manage routing, use client-side templates, etc. You are going to have to write a fair bit of Javascript to get things rolling.

Depending on how you implement things, you are going to probably have a bunch of server side logic lying around as well.

With a PVC framework, on the other hand, keeps the majority of your logic on the server side. Depending on the framework, you end up writing only a bit more code than you would in a more traditional web application and, since everything is still rendered server side, you eliminate the need for heavy client-side work.

Why would I use Intercooler?

Both TurboLinks and pjax are targeted PVC frameworks: they are mostly concerned with navigation.

Intercooler is a more complete PVC framework: there are patterns for updating things as small as a progress bar width, for polling, for passing actions to the server, for working with tables/bulk operations, refreshing specific elements in the DOM, etc.

One way to think about Intercooler is as an Angular alternative, with REST-ful bindings to a server side model & renderer.

And a lot less code.

What are the core concepts?

The core concepts in Intercooler are:

  • HTML Elements are declaratively bound to REST-ful HTTP endpoints via the standard HTTP verbs (GET, POST, etc.)
  • Dependencies between elements are encoded in their REST-ful path bindings
  • Server responses are HTML fragments that will be swapped in place of an existing elemtents content
  • Server responses can have HTTP header directives as well, enabling fine-grain control over the UI

I'm skeptical.

Sure, as you should be. This is an as-yet uncodified approach to building web applications. However, many successful web applications are already being built this way. Basecamp, for example, is a famous one. It doesn't use Intercooler, of course, but it does use the PVC-style.

And this approach isn't right for every UI: if you have a bunch of stuff flying around on your UI and lots of user input patterns, Intercooler isn't the right thing. However, even in applications like that you probably have some plain old form or table UI's sitting around for which a full Javascript MVC framework is overkill. There, Intercooler can help.
